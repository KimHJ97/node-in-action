# 마이크로 태스크 큐

## 1. Microtask Queue

콜백 큐는 하나가 아니다. 자바스크립트 엔진 내부에는 우선순위가 다른 두 가지 대기실이 존재한다.

Microtask Queue는 비동기 작업 중에서도 `더 시급하고 중요한 작업`을 처리하기 위한 전용 대기실이다. 일반적인 비동기 작업(Task)보다 우선순위가 훨씬 높아서, 이벤트 루프가 가장 먼저 달려가서 확인하는 곳이다.

- `Promise`: `.then()`, `.catch()`, `.finally()` 콜백 함수
- `async` / `await`: await 이후의 코드들
- `MutationObserver`: DOM 변경을 감지하는 API
- `queueMicrotask()`: 명시적으로 마이크로태스크를 예약하는 함수

### 1-1. 우선순위 비교 (Task vs Microtask)

이벤트 루프는 Microtask Queue가 비어있다면, 그 이후에 일반 Task Queue를 확인한다.

마이크로 태스크를 처리하는 도중에 새로운 마이크로태스크가 생기면, 그것까지 다 끝내야만 다음 단계로 넘어간다. 즉, 마이크로태스크 안에서 계속해서 새로운 마이크로태스크를 생성하면, 일반 `setTimeout`은 영원히 실행되지 못하고 브라우저가 멈춰버릴 수 있다.

- **태스크 큐 (Macrotask)**
  - 예시: `setTimeout`, `setInterval`, `UI Rendering`
  - 우선순위: 낮음
  - 처리 방식: 이벤트 루프가 한 번에 하나씩
- **마이크로태스크 큐 (Microtask)**
  - 예시: `Promise`, `async`/`await`, `queueMicrotask`
  - 우선순위: 높음
  - 처리 방식: 이벤트 루프가 큐에 있는 모든 작업을 싹 비움

```javascript
// 1. 시작 -> 4. 끝 -> 3. 프로미스 -> 2. 타임아웃
console.log("1. 시작 (Sync)");

setTimeout(() => {
  console.log("2. 타임아웃 (Macrotask)");
}, 0);

Promise.resolve().then(() => {
  console.log("3. 프로미스 (Microtask)");
});

console.log("4. 끝 (Sync)");
```

## 2. Microtask Queue의 필요성

- **데이터의 일관성 보장 (Immediate State Consistency)**
  - 비동기 작업의 결과가 나왔을 때, 그 결과를 "브라우저가 다른 짓(렌더링, 사용자 클릭 등)을 하기 전에" 즉시 반영하기 위해서입니다.
  - 문제 상황: 만약 Promise가 일반 태스크 큐(setTimeout)와 똑같이 취급된다면, API 응답이 왔을 때 브라우저가 갑자기 화면을 새로 그리거나 사용자의 클릭 이벤트를 먼저 처리해버릴 수 있습니다.
  - 해결: 마이크로태스크 큐를 쓰면 현재 실행 중인 자바스크립트가 끝나자마자(스택이 비자마자) 바로 결과를 처리하므로, 사용자가 중간의 어정쩡한 상태(데이터가 없는 화면 등)를 볼 틈이 생기지 않습니다.
- **효율적인 배치 처리 (Batching for Performance)**
  - 여러 번 일어날 변화를 한 번에 모아서 처리하여 성능을 높이기 위함입니다. 현대 프런트엔드 프레임워크(React, Vue 등)가 이 원리를 아주 잘 활용합니다.
  - 예시 (데이터 업데이트): 1초 안에 데이터가 100번 바뀐다고 가정해 봅시다.
    - 매번 렌더링하면: 브라우저가 100번 화면을 그리느라 엄청나게 느려집니다.
    - 마이크로태스크 활용: 데이터가 바뀔 때마다 마이크로태스크 큐에 "나중에 화면 그려!"라고 예약만 해둡니다. 자바스크립트 실행이 끝난 직후(마이크로태스크 단계)에 100번째 최종 결과만 딱 한 번 화면에 그립니다.
  - 필요성: 사용자에게는 결과만 보여주면 되므로 불필요한 연산 낭비를 획기적으로 줄여줍니다.

## 3. Microtask Queue의 중단

await를 만나면, 실행중인 MicroTask 실행은 종료된다. 그 사이에 MacroTask 실행 기회가 생긴다. 그 다음 Then 콜백은 다시 MicroTask 큐에 들어간다.

- `await` 뒤에 작업이 완료(Resolve)되면, 함수의 나머지 부분이 Microtask Queue에 들어간다.

```javascript
// 1. await 사용
// A -> Timeout -> B
function delay(ms) {
  return new Promise((resolve) => setTImeout(resolve, ms));
}

Promise.resolve()
  .then(async () => {
    console.log("A");
    await delay(2000);
  })
  .then(() => {
    console.log("B");
  });

setTimeout(() => {
  console.log("Timeout");
}, 1000);

// 2. Promise.resolve()만 사용
// A -> B -> Timeout
Promise.resolve()
  .then(() => {
    console.log("A");
    return Promise.resolve();
  })
  .then(() => {
    console.log("B");
  });

setTimeout(() => {
  console.log("Timeout");
}, 0);
```

## 4. MutationObserver

`MutationObserver`는 `DOM(웹 페이지의 구조)에 변화`가 생겼을 때 이를 감지하여 특정 로직을 실행하는 도구이다.

- 웹 페이지에서 요소가 추가되거나, 속성(class, id 등)이 바뀌거나, 텍스트 내용이 변경될 때 이를 실시간으로 관찰한다.
- 과거에 사용하던 MutationEvents의 성능 문제를 해결하기 위해 등장한 현대적인 API이다.

```html
<div id="box">Box</div>
<button id="btn">Add</button>

<script>
  const box = document.getElementById("box");
  const btn = document.getElementById("btn");

  // DOM changed: -> setTimeout
  const observer = new MutationObserver((mutations) => {
    console.log("DOM changed: ", mutations);
  });

  observer.observe(box, { childList: true });

  btn.addEventListener("click", () => {
    const newEl = document.createElement("p");
    newEl.textContent = "New element added";
    setTimeout(() => {
      console.log("setTimeout");
    }, 0);
    box.appendChild(newEl);
  });
</script>
```
